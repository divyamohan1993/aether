Aether Documentation

Edge AI for disaster response. A 40 KB SOS payload, no Speech-to-Text, Gemini 2.5 Flash hearing the panic, the dialect, and the river behind the door, returning a strict-schema dispatch brief in under two seconds.

This portal explains every public-facing surface of Project Aether: the Survivor PWA at /, the Task Manager at /tm/, the scope hierarchy that runs from a national NDMA seat down to a single field volunteer, the post-quantum auth that makes user impersonation impossible, and every HTTP endpoint the service speaks.

Pick a role below to jump straight to the part you need.

02Survivor portal

A pure-black, hold-to-record SOS button. Built for a phone with one battery bar and one signal bar. The whole page is 12 KB raw, 4.5 KB brotli, served once and cached by the service worker.

What happens when the button is pressed

  1. The browser asks for microphone access. The mic icon is the only permission this page ever asks for.
  2. MediaRecorder records Opus audio at 12 kbps mono, 16 kHz. Fifteen seconds of speech compresses to roughly 40 KB.
  3. If the device is offline, the clip goes into IndexedDB. A queue badge in the top bar shows how many clips are waiting.
  4. The next time the OS reports a network handshake, the queue flushes. Background Sync is registered when supported.
  5. The clip is POSTed to /api/v1/triage with optional headers for language, geo, battery, and effective network type.
  6. The Cloud Run server forwards the audio to Vertex AI Gemini 2.5 Flash with a strict responseSchema.
  7. Gemini hears the audio natively. No Speech-to-Text in the middle. It returns urgency, native and English transcription, casualties, needs, location clues, ambient cues, summary, confidence.
  8. The PWA renders an urgency badge and a one-line dispatch summary. The survivor sees a calm message: help is on the way, stay where you are.

Languages

Auto-detected from navigator.language. Native-script strings ship for:

  • en-IN English (India)
  • hi-IN Hindi
  • ta-IN Tamil
  • bn-IN Bengali

Gemini detects the spoken language from the audio itself. The browser hint is a tie-breaker, never a hard constraint.

Battery and network awareness

  • Background is #000000. On OLED screens, those pixels are physically off and draw zero current.
  • The top bar shows battery percentage (orange below 15%), the connection effective type from navigator.connection, queue depth, and language.
  • No polling. The page reacts to online, offline, and connectionchange events. The service worker uses background sync where the platform supports it, otherwise an exponential retry loop.
  • The record button works from the keyboard (Space or Enter) and from media-session keys.

Privacy

  • What is sent: the Opus clip, a UUIDv4 client ID stored in localStorage, language hint, optional rough geolocation, optional battery and network type, an ISO 8601 timestamp.
  • What is stored locally: just the queue and the client ID. Nothing else.
  • Retention: server-side audio is processed in memory by Cloud Run, sent to Vertex AI, and discarded. No durable copy of the audio is kept by Aether. Triage JSON is logged with request ID, urgency, and confidence for operations.
  • SDRF webhook delivery shape: the same JSON document the API returns, signed with the service's HMAC and forwarded to the configured webhook URL. The webhook is opt-in per state.

03Task Manager portal

A post-quantum-authenticated task tracker for the NDMA chain of command. Every privileged action carries an ML-DSA-65 signature. The server cannot impersonate a user, even with full database access.

Mounted at /tm/. Single-page app, hash-routed, four sections: dashboard, projects, tasks, users.

Login flow Skip

Two fields: email and passphrase.

  1. You type your email and click Continue. The browser POSTs { email } to /api/v1/tm/auth/challenge. The server returns a 32-byte random challenge, even if the email is unknown (so an attacker cannot enumerate accounts).
  2. You type your passphrase. The browser pulls the encrypted private key from IndexedDB, derives a 32-byte key with Argon2id, and decrypts it with AES-256-GCM. The passphrase never leaves the device.
  3. The browser signs the challenge with your ML-DSA-65 private key and POSTs { email, ch_id, signature_b64 } to /api/v1/tm/auth/verify.
  4. The server fetches your stored public key, runs ml_dsa65.verify, and on success returns a session token signed by its own ML-DSA-65 keypair.
  5. The token lives in sessionStorage for thirty minutes. Privileged actions still need a fresh per-action signature on top of the bearer.

Register flow Skip

  1. An admin one tier above you mints an invite. You receive an invite link with an opaque invite_id.
  2. You open the link, type your name, and choose a passphrase. The browser generates a fresh ML-DSA-65 keypair locally.
  3. The private key is encrypted with Argon2id + AES-256-GCM and written to IndexedDB.
  4. The browser sends { invite_id, name, pubkey_b64 } to /api/v1/tm/auth/register.
  5. The server checks the invite signature, marks it used in a Firestore transaction, writes your user record, and returns your fresh session token.
Lose the passphrase, lose the seat. The server never sees your passphrase or your private key. Recovery is by re-invitation from your parent in the hierarchy.

Dashboard Skip

Tile counts at the top, scoped to your subtree:

  • Open, In progress, Blocked, Done, Overdue, Cancelled.
  • Priority breakdown: low, medium, high, critical.
  • Top assignees, ranked by open task count.
  • Projects active.

Backed by GET /api/v1/tm/dashboard. Counts are filtered by your scope_path, so an SDMA officer in Himachal Pradesh sees Himachal numbers only.

Projects Skip

  • List. Scope-filtered. ?include_archived=true to see archived ones.
  • Create. Name, description, owner, scope. The scope must sit inside your own.
  • Drill-down. Click a project to see its tasks, assignees, and progress.
  • Archive. Requires a fresh ML-DSA signature. The browser pops the passphrase dialog. Archive is reversible only by editing the document directly.

Tasks Skip

  • Create. Project, title, assignee, priority, due date.
  • Assign. Reassignment requires a fresh signature on the canonical message task.assign|<tid>|<assignee_uid>.
  • Status transitions: open, in progress, blocked, done, cancelled. The server validates legal transitions.
  • Priority: low, medium, high, critical.
  • Due dates. The dashboard counts a task as overdue when due date is past and status is not done or cancelled.
  • Volunteer rule: status-only updates on tasks they are assigned to. Anything else returns 403.
  • Responder rule: full edits on tasks they are assigned to.
  • District and above: edit any task whose scope_path sits inside their own.

Users Skip

  • Hierarchy table, sorted by tier and email. Each row shows tier, scope, parent, and active state.
  • Two-click delegation. Click a user, click a new tier. The dialog reads back the change in plain language. Submit triggers the passphrase prompt for a fresh signature.
  • The server validates: the new tier is at or below the caller's tier, the target's scope_path sits inside the caller's, the target is currently below the caller. Rejected delegations write a warning to the audit log too.
  • Suspend. Disables a user without deleting them. They can be reactivated by the same parent.
  • Every mutation appends to tm_audit with the canonical message and the signature, so post-incident review can replay who did what.

Logout

Clicking Log out clears the session token from sessionStorage. The encrypted private key stays in IndexedDB so the next login on the same device only needs the passphrase. To wipe the device, clear site data from the browser settings.

04Hierarchy and scopes

Every user, project, and task carries a scope_path. Scope is a slash-separated lineage, and authority flows down the path. A user can act on any record whose scope starts with their own.

TierCodeScope shapeAuthority
ndma100ndmaNational. Reads everything. Delegates to anyone.
state80ndma/<state>One state's subtree.
district60ndma/<state>/<district>One district's subtree.
responder40.../<responder>Field staff. Owns assigned tasks.
volunteer20.../<volunteer>Read-only on assigned tasks. Status-only updates.

Worked examples

  • An NDMA officer at ndma can read every user, every project, every task. Their queries are unfiltered.
  • An SDMA officer at ndma/HP can read users with scope ndma/HP, ndma/HP/Shimla, ndma/HP/Kullu/responder1, but not ndma/UK.
  • A responder at ndma/HP/Shimla/responder7 can update only the tasks where they are the assignee. They cannot read other responders' tasks even within Shimla.
  • A volunteer at ndma/HP/Shimla/responder7/volunteer3 can flip the status of an assigned task from open to in progress to done. They cannot change priority, assignee, or due date.

Two-click delegation, walked through

  1. You are SDMA Himachal at ndma/HP. You open the users page.
  2. You click a row for a district officer at ndma/HP/Mandi. Their current tier reads as 60.
  3. You pick a new tier (say 40, demoting them to responder during a leave). The dialog reads: "Set Mandi DDMO to tier 40 (responder)?".
  4. You confirm. The browser asks for your passphrase, signs the canonical message delegate|<uid>|40, and posts it.
  5. The server checks: 40 is at or below your tier (true), the target sits in your subtree (true). It writes the change inside a Firestore transaction along with an entry in tm_audit.
  6. The user list refreshes. The change is visible to anyone in scope within seconds.

05API reference

Every endpoint speaks JSON. Errors share one shape: { "error": { "code": "...", "message": "..." } }. Every response carries a request ID in the X-Request-Id header.

Group 1. SOS Triage

POST /api/v1/triage public

Accepts an audio body up to 512 KB. The first bytes are sniffed; only Opus/Ogg, WebM, MP4, and WAV are accepted.

Request headers

Content-Type
audio/webm, audio/ogg, audio/mp4, or audio/wav.
X-Client-Id
UUIDv4 persisted on the device. Optional.
X-Client-Lang
BCP-47 hint such as hi-IN. Optional.
X-Client-Geo
Best-effort lat,lng. Optional.
X-Client-Battery
Integer 0 to 100. Optional.
X-Client-Network
2g, 3g, 4g, or wifi. Optional.
X-Client-Timestamp
ISO 8601 from the device. Optional.

  "latency_ms": 1777,

    "urgency": "CRITICAL|HIGH|MEDIUM|LOW|UNCLEAR",

Errors

  • 400 empty_body, 400 bad_request
  • 405 method_not_allowed

  • 429 rate_limited with Retry-After
  • 502 upstream_error (Vertex AI unavailable)

GET /api/healthz public

Returns { "ok": true } with status 200. Used by Cloud Run health checks and uptime probes.

Group 2. Task Manager

GET /api/v1/tm/server-pubkey public

Returns the server's ML-DSA-65 public key in base64, plus the algorithm name. Used by the browser to verify session tokens offline.

{ "pubkey_b64": "...", "alg": "ML-DSA-65" }
curl -s https://aether.dmj.one/api/v1/tm/server-pubkey
POST /api/v1/tm/auth/challenge public

Body: { "email": "..." }. Returns a 32-byte challenge bound to the email and a one-minute TTL.

{ "ch_id": "...", "challenge_b64": "...", "expires_at": "..." }

Errors: 400 bad_request, 429 rate_limited.

curl -X POST https://aether.dmj.one/api/v1/tm/auth/challenge \
  -H "Content-Type: application/json" -d '{"email":"ops@example.com"}'
POST /api/v1/tm/auth/verify public

Body: { "email", "ch_id", "signature_b64" }. Consumes the challenge in a Firestore transaction so it cannot be replayed.

{ "token": "<ML-DSA session token>", "exp": "...", "user": { ... } }

Errors: 400 bad_request, 401 invalid_signature, 403 user_suspended, 410 challenge_expired, 429 rate_limited.

POST /api/v1/tm/auth/register public

Body: { "invite_id", "name", "pubkey_b64" }. Server verifies the issuer's signature on the invite, then creates the user atomically with the invite-used flag flip.

Errors: 400 bad_request, 404 invite_not_found, 409 invite_used, 410 invite_expired.

POST /api/v1/tm/auth/bootstrap public env-gated

One-time NDMA root creation. Active only when TM_BOOTSTRAP_ALLOW=1 and a matching TM_BOOTSTRAP_TOKEN is present. See Operations for the procedure.

GET /api/v1/tm/me bearer

Returns the actor record: uid, email, tier, tier_name, scope_path, parent_uid, active.

curl -H "Authorization: Bearer $TOKEN" https://aether.dmj.one/api/v1/tm/me
GET /api/v1/tm/users bearer

Scope-filtered list. Query: ?scope=<scope_path>&limit=100. Defaults to the actor's own scope.

Errors: 400 bad_scope, 403 out_of_scope.

GET /api/v1/tm/users/:uid bearer

Single user. Returns 403 if the target sits outside the caller's scope.

POST /api/v1/tm/users/invite bearer + fresh sig

Body: { "email", "tier", "parent_uid", "scope_path", "signature_b64" }. Caller's signature must cover invite|<email>|<tier>|<parent_uid>|<scope_path>.

Errors: 400 bad_tier, 400 bad_scope, 403 tier_too_high, 403 out_of_scope, 409 already_invited.

POST /api/v1/tm/users/:uid/delegate bearer + fresh sig

Body: { "new_tier", "signature_b64" }. Signature covers delegate|<uid>|<new_tier>.

Errors: 400 bad_tier, 403 tier_too_high, 403 out_of_scope, 403 tier_at_or_above_actor.

POST /api/v1/tm/users/:uid/suspend bearer

Body: { "suspend": true|false }. Disables or reactivates a user.

GET POST /api/v1/tm/projects bearer

List or create. GET takes ?include_archived=true&limit=100. POST body: { "name", "description", "scope_path" }.

GET PATCH DELETE /api/v1/tm/projects/:pid bearer

GET returns the project. PATCH updates name, description, or status. DELETE archives the project and requires a fresh signature on project.archive|<pid>.

GET POST /api/v1/tm/tasks bearer

GET filters: ?project=<pid>&assignee=<uid>&status=open&overdue=true&limit=100. POST body: { "project_id", "title", "assignee_uid", "priority", "due_at" }.

GET PATCH DELETE /api/v1/tm/tasks/:tid bearer

Volunteers can only PATCH { "status" } on tasks they are assigned to. Reassigning the task ({ "assignee_uid", "signature_b64" }) requires a fresh signature on task.assign|<tid>|<assignee_uid>.

GET /api/v1/tm/dashboard bearer

Aggregates by status, priority, top assignees, projects active, scoped to the caller. Cached server-side per actor for ten seconds.

curl -H "Authorization: Bearer $TOKEN" https://aether.dmj.one/api/v1/tm/dashboard

06Security model

Why post-quantum, why now

ML-DSA-65 is the digital signature scheme standardised by NIST as FIPS 204. It is built on lattice problems that resist both classical and quantum attacks. A 2030-era adversary recording today's session traffic gains nothing from breaking it later, because no recovery secret ever leaves the user's device.

Each TM user has their own keypair, generated in the browser at registration. The server never sees the private key. Login is challenge-response. Privileged actions carry a per-action signature whose canonical message names the action and target, so it cannot be replayed against a different record.

Private key custody

  • Generated in the browser via @noble/post-quantum.
  • Encrypted at rest in IndexedDB using AES-256-GCM. Key derivation is Argon2id with t=3, m=64 MiB, p=1, dkLen=32.
  • The passphrase is the only thing held in human memory. The server cannot brute-force it because it never sees the encrypted blob.

Server keypair custody

  • The server's ML-DSA-65 keypair lives in Google Secret Manager.
  • Cloud Run mounts the secret via a least-privilege binding to the runtime service account.
  • Rotation is a swap of the secret version plus a redeploy. Old session tokens are invalidated within their thirty-minute TTL.

Transport and headers

  • HSTS preload: max-age=63072000; includeSubDomains; preload.
  • Strict CSP: default-src 'self'; connect-src 'self'; frame-ancestors 'none'; base-uri 'none'. No third-party scripts, no inline frames.
  • Permissions-Policy: locked to first-party microphone and geolocation only. Camera, USB, payment, MIDI, and friends are off.
  • X-Content-Type-Options: nosniff, Referrer-Policy: no-referrer, Cross-Origin-Opener-Policy: same-origin, Cross-Origin-Resource-Policy: same-origin.

Rate limits

  • Triage: 10 requests per minute, 200 per day, per IP. Token-bucket, in-memory, resets on cold start.
  • TM auth: 30 requests per minute, 200 per day, per IP. Same shape.
  • Both return 429 rate_limited with a Retry-After header.

Compliance

  • DPDP Act 2023 alignment: explicit consent for microphone use, granular and revocable from the device's site settings, audio retained only for processing, no PII in logs.
  • GDPR alignment: data subject access via GET /api/v1/tm/me, deletion via the parent in the hierarchy, data minimisation by design.
  • India region pinning: Firestore database is Indian-region (asia-south2); Cloud Run can be re-deployed to asia-south1 for production.

07Operations

Cost shape

$0 idle by design. The whole stack scales to zero when no one is using it.

  • Cloud Run: min-instances=0. Charged only for in-flight request seconds.
  • Vertex AI: pay-per-token; idle cost is zero.
  • Firestore Native: free tier covers 1 GB stored, 50K reads, 20K writes per day, more than enough for an MVP.
  • Secret Manager: two secret versions, well inside the six-version free tier.
  • Artifact Registry: image well under the 0.5 GB free tier.
  • Cloud Build: under the 120 build-minute daily free tier.

Bootstrap procedure (NDMA root)

  1. Generate the operator's ML-DSA-65 keypair locally with the script in the repo. Save pub.b64 and priv.b64.
  2. Set the gate flag: gcloud run services update aether --update-env-vars=TM_BOOTSTRAP_ALLOW=1.
  3. POST { email, name, pubkey_b64 } to /api/v1/tm/auth/bootstrap.
  4. Unset the flag: --update-env-vars=TM_BOOTSTRAP_ALLOW=0.
  5. Keep priv.b64 offline. Lose it and the root account is unrecoverable by design.

Adding the first user at each tier

  1. The NDMA root signs in and opens /tm/#/users.
  2. Invite a state officer with their email, tier 80, scope ndma/<state>, parent set to the NDMA root. Share the invite link.
  3. The state officer signs up. Repeat the invite step for a district at tier 60, scope ndma/<state>/<district>.
  4. Repeat down the chain to responder (40) and volunteer (20).

Logs and observability

  • Every request emits structured JSON to Cloud Run stdout. Each line carries a timestamp, severity, function, request ID, IP, and the action-specific fields.
  • Cloud Logging is the search surface. Filter by jsonPayload.requestId to follow a request across the triage and TM modules.
  • Audit log tm_audit is append-only and stores the canonical signed message for every privileged action.
  • X-Request-Id is echoed on every HTTP response.

Backup and restore

  • Firestore export to a Cloud Storage bucket via gcloud firestore export. Schedule daily.
  • Secret Manager versions are immutable. A rotation creates a new version; old ones can be disabled but kept for forensic recovery.
  • The Survivor PWA is a static bundle. Restore is a redeploy; no client state to recover.

09FAQ

Why no passwords?
Passwords have to be sent to a server, hashed, compared, and stored. Each step is a place to leak them. ML-DSA-65 challenge-response means the server never sees a secret. Compromising the database does not compromise the users.
Can I lose my account if I forget my passphrase?
Yes. The passphrase decrypts your private key in the browser. Without it the key is unrecoverable. Recovery is by re-invitation from your parent in the hierarchy. This is on purpose: it means an operator with database access still cannot impersonate you.
How is this 2G-friendly?
The Survivor PWA is 12 KB raw, 4.5 KB brotli. The audio is Opus at 12 kbps, around 40 KB for a 15-second clip. A 1.5-second window of usable signal is enough to ship the whole payload.
Why Gemini and not Whisper?
Whisper transcribes speech. In a panicked, multilingual, wind-and-static scenario it tends to drop or mangle words. Gemini's native audio modality reads tone, dialect mix, and ambient cues directly, then outputs JSON in one call. Fewer stages, fewer places to fail.
Why post-quantum already?
Disaster-response audit trails outlive the device that signed them. A signature today must still be trustworthy in 2040. ML-DSA-65 is the NIST-standardised choice that meets that bar without exotic primitives.
What if Cloud Run cold-starts during a disaster?
Cold start for the Aether image is around 800 ms on a free-tier instance. The PWA's IndexedDB queue covers any failed POSTs and retries on reconnect. For a paid pilot, set min-instances=1 in the affected region for sub-100 ms p99 first response.
How do I add a state-level user?
NDMA root signs in, opens the users page, invites the state officer with tier 80 and scope ndma/<state>. The state officer signs up using the invite link. See Operations for the full chain.
Where does my voice recording go?
From the browser, encrypted in transit by TLS 1.3, into the Cloud Run process memory, then into Vertex AI for Gemini to listen to. The Cloud Run container does not persist audio. After the response is sent, the buffer is freed. Triage JSON is logged with request ID, urgency, and confidence; not the audio, not the transcript.
What runs offline?
The Survivor PWA shell, the queue, the language strings, the privacy notice. The triage call itself needs network, but the queue waits for it without losing the clip.
Is it production-ready?
It is a 24-hour assessment build: end-to-end working, hardened headers, rate limits, scope checks, audit log. Production hardening for a state pilot would add a webhook signing key, Firestore backups, an in-region pin to asia-south1, and a synthetic uptime probe.
